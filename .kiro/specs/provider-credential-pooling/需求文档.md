# 预置供应商凭证池化需求文档

## 背景

ArcReel 的预置供应商设置页已经支持为同一个供应商保存多条凭证，但运行时仍采用单一 `active` 凭证。其他凭证只能作为手动切换备用，不能自动参与生成任务。

Agnes 免费层级下，用户会配置多条来自不同账号的 key。每个账号/key 同一时间只能执行 1 个生成任务，并且图片与视频共享这个并发名额。当前单 `active` 模型无法把任务分配到多个账号，也无法保证 `key-1` 提交的 provider job 后续仍由 `key-1` 查询。

## 目标

为每个预置供应商增加独立的凭证池化能力。供应商开启池化后，新生成任务从该供应商已启用且空闲的凭证中租赁一条；任务提交后绑定该凭证，后续轮询、恢复、下载都固定使用同一条凭证。

未开启池化的供应商必须保持现有行为：只使用当前 `active` 凭证。

## 非目标

- 不改变自定义供应商的凭证模型，除非后续明确纳入范围。
- 不改写供应商模型选择、价格计算、用量统计的业务语义。
- 第一版不要求按供应商内置默认并发策略自动识别免费/付费层级。
- 第一版不要求凭证级复杂限流策略，如 RPM、权重、429 冷却、额度感知。
- 第一版不要求新增 provider 级总并发配置；现有 `image_max_workers` / `video_max_workers` 保持原语义。

## 核心概念

### 池化开关

每个预置供应商拥有独立的 `credential_pool_enabled` 开关。

- 关闭：生成任务只使用该供应商的 `active` 凭证，完全兼容当前行为。
- 开启：生成任务从该供应商参与池化的凭证中租赁空闲凭证。

不同供应商互不影响。Agnes 可以开启池化，OpenAI 保持单 `active`，Vidu 单独开启或关闭。

### 参与池化

每条凭证拥有是否参与池化的标记，例如 `is_enabled`。

- `is_enabled=true`：当所属供应商开启池化时，该凭证可被新任务租赁。
- `is_enabled=false`：不接新任务。
- 禁用凭证不影响已租赁或已绑定到 provider job 的任务。

现有 `is_active` 继续保留，用于：

- 池化关闭时的唯一运行凭证。
- 连接测试默认凭证。
- 兼容旧数据和旧流程。

### 租赁

租赁是任务执行前对某条凭证的占用。租赁必须具备原子性，避免两个并发任务同时租到同一条凭证。

任务租赁成功后，生成链路必须使用该凭证构造 backend。任务终态后释放租赁。

### 任务绑定

任务提交 provider job 后，必须持久化：

- ArcReel `task_id`
- `provider_id`
- provider 侧 `provider_job_id`
- ArcReel `credential_id`
- `media_type`
- `model_id` 或等价模型标识

后续 poll、resume、download 必须使用绑定的 `credential_id`，不得重新从池中选择凭证，也不得使用当前 `active` 凭证兜底。

## 用户故事

1. 作为创作者，我可以为 Agnes 添加多个不同账号的 key，并开启 Agnes 的凭证池化，让多个任务自动分配到不同账号上。
2. 作为创作者，我可以只给 Agnes 开启池化，而让 OpenAI、Vidu 等供应商继续使用当前单 `active` key 行为。
3. 作为创作者，我可以禁用某条参与池化的 key，使它不再接新任务，但不打断已经提交给 provider 的任务。
4. 作为创作者，当所有 key 都忙时，我希望后续任务排队等待可用 key，而不是失败或使用错误 key。
5. 作为维护者，我可以在 worker 重启后恢复视频任务，并保证恢复轮询使用提交该任务时的原始 key。
6. 作为维护者，我希望删除 key 时系统阻止删除仍有关联运行中任务的凭证，避免任务无法恢复。

## 功能需求

### FR-1 Provider 级池化开关

每个预置供应商可单独开启或关闭凭证池化。

- 默认关闭。
- 升级后老用户行为不变。
- 关闭时不读取租赁池，不改变当前 active key 调用路径。
- 开启时只影响该 provider 的新任务调度。

### FR-2 凭证参与池化标记

每条预置供应商凭证可设置是否参与池化。

- 池化关闭时，该标记不影响生成行为。
- 池化开启时，只有同 provider 且 `is_enabled=true` 的凭证可被租赁。
- 新增凭证的默认参与策略需要保守，建议默认不参与池化，或在池化已开启时由 UI 明确提示用户选择。

### FR-2.1 池化 UI 状态切换

凭证列表 UI 需要随 provider 的池化开关切换展示语义，避免用户把 `active` 误解为池化开启后的唯一生效凭证。

池化关闭时：

- 保持当前单选 UI。
- 每个 provider 最多一条凭证显示为“使用中”。
- 点击未激活凭证的圆点仍表示切换 active 凭证。
- 生成任务只使用 active 凭证。

池化开启时：

- 凭证行不再用单选圆点表达“唯一使用中”。
- 每条凭证显示“参与池化”开关，对应 `is_enabled`。
- `active` 凭证可保留为“默认”或“首选”标记，用于连接测试默认项和关闭池化后的回退，但不表示池化开启时只有它会生成。
- 若没有任何凭证参与池化，应在凭证列表或 provider 详情中提示“当前没有可用池化凭证”。
- 切换池化开关后，应刷新 provider 配置状态与凭证列表，避免旧 UI 状态残留。

池化关闭但仍有已租赁/已绑定任务运行时：

- 关闭池化只影响关闭后的新任务。
- 已租赁或已绑定任务继续使用原 `credential_id` 直到终态。
- UI 应在任务详情或凭证区域展示“仍有任务使用池化凭证运行中”，或至少在任务详情展示实际凭证名称/标识，避免用户误以为运行中任务已切换到 active 凭证。

池化开关、参与池化开关、并发模式切换都必须按幂等 PATCH 语义实现。前端保存期间应禁用对应控件；保存失败时回滚 UI 或重新拉取后端状态。

### FR-3 租赁模式

池化开启时，任务在 submit 前必须先租赁凭证。

- 租赁成功：任务使用该凭证构造 backend 并继续执行。
- 租赁失败：任务不得 submit，不得失败，不得回退 active key；任务保持等待，稍后重新调度。
- 租赁必须按 provider 隔离，不同 provider 的池互不影响。
- 租赁池是平台调度层能力，不是 provider backend 能力。各 provider backend 不应感知 `credential_pool_enabled`、lease、`is_enabled`；它们只接收已由 `credential_id` overlay 后的配置。新增预置供应商只要走统一 `assemble_backend` / 指定凭证配置解析路径，就应自动获得池化能力。

### FR-4 并发模式

凭证池支持两种 provider 级并发模式：

- `shared`：图片和视频共享同一凭证并发名额。
- `separate`：图片和视频分别计算凭证并发名额。

第一版默认推荐 `shared`，更适合 Agnes 免费层级。

`shared` 行为：

- 一条 key 正在跑图片任务时，视频任务不能租赁该 key。
- 一条 key 正在跑视频任务时，图片任务不能租赁该 key。

`separate` 行为：

- 一条 key 正在跑图片任务时，视频任务仍可租赁该 key。
- 一条 key 正在跑视频任务时，图片任务仍可租赁该 key。
- 同一 media lane 的第二个任务不能超过该 lane 的凭证并发名额。

第一版可将每条凭证的默认容量设为：

- `shared`：`total_max_inflight=1`
- `separate`：`image_max_inflight=1`，`video_max_inflight=1`

### FR-5 任务绑定与恢复

视频 submit 成功后，系统必须持久化 provider job 与 credential 的绑定。

- 当前进程内 poll 使用同一 backend。
- worker 重启后的 resume 根据绑定的 `credential_id` 重新构造 backend。
- 若旧任务没有绑定 `credential_id`，只能走历史兼容路径，使用 active 凭证并记录警告。

### FR-6 Backend 缓存隔离

池化开启时，backend cache key 必须包含 `credential_id`。

示例：

```text
(media_type, provider_id, model_id, credential_id)
```

池化关闭时可继续使用兼容 key，或将 `credential_id=None` 纳入统一 cache key。

### FR-7 等待行为

当某 provider 开启池化且无空闲凭证时：

- 任务继续等待，不标记失败。
- 任务不得 submit。
- 任务不得切换 provider。
- 任务不得使用已被租赁的凭证。
- 前端应尽量展示“等待可用凭证”之类的原因。

第一版可复用现有 queued/running 状态，但需要记录等待原因，以便 UI 或日志可见。

凭证可用性必须作为 worker claim 前的调度过滤条件，而不是任务执行内部的长等待点。目标行为：

```text
claim cycle
  -> provider lane has room?
  -> credential pool has idle lease candidate?
  -> claim task
  -> acquire credential lease atomically
  -> register provider slot
  -> execute submit/poll
  -> terminal -> release provider slot + credential lease
```

当 provider 开启池化且无可租赁凭证时，worker 应在 claim 前跳过该 provider 的任务，使任务保持 `queued`。不得先 claim 为 `running`、注册 `SlotTable` 占用后再等待凭证。若因竞态在 claim 后租赁失败，必须立即把任务 requeue，并把该 provider 加入本轮 credential-full 黑名单，避免同一轮反复 claim/requeue。

### FR-8 删除与禁用

禁用凭证：

- 阻止新任务租赁。
- 不影响已租赁或已绑定任务。

删除凭证：

- 如果存在 active lease、running task、pending resume task 或未终态 provider job 绑定该凭证，拒绝删除。
- 删除失败应返回用户可理解的错误文案。

### FR-9 连接测试

连接测试保留按指定凭证测试的能力。

- 测试单条凭证：使用该 `credential_id`。
- 测试供应商默认：池化关闭或未指定凭证时使用 `active` 凭证。
- 可选增强：池化开启时提供“测试池中所有凭证”的入口。

### FR-10 错误与救援登记表

池化相关等待原因、失败原因和接口错误必须使用稳定 code，不直接持久化中文文案。用户可见文案必须走 i18n。任务失败类原因应优先复用或扩展 `lib/task_failure.py` 的结构化失败编码；非失败等待类原因可存入 task metadata / wait reason，但同样必须是稳定字符串。

| 场景 | 建议代码/标识 | 处理方式 | 用户看到 |
| --- | --- | --- | --- |
| 池化开启但无 enabled 凭证 | `waiting_for_credential` | 不 claim / 保持 queued | 等待可用凭证 |
| 所有凭证已被租赁 | `waiting_for_credential` | 不 claim / 保持 queued | 等待可用凭证 |
| claim 后竞态导致租赁失败 | `credential_lease_conflict` | requeue，本轮 provider 加入 credential-full 黑名单 | 仍显示等待可用凭证 |
| provider job 已 submit 但绑定 `credential_id` 持久化失败 | `credential_binding_persist_failed` | fail-fast，避免不可恢复幽灵任务 | 任务失败，提示绑定凭证失败 |
| resume 时旧任务缺少 `credential_id` 绑定 | `credential_binding_missing` | 旧任务兼容 active，记录 warning | 一般不提示；失败时显示原失败原因 |
| 删除仍被任务占用的凭证 | `credential_in_use` | HTTP 409，拒绝删除 | 凭证仍有关联运行中任务，无法删除 |
| 释放 lease 失败 | `credential_lease_release_failed` | 重试；仍失败则记录告警，并由恢复扫描修复 | 一般不提示 |

`credential_binding_persist_failed` 属于提交后关键持久化失败。该路径不得静默继续 poll，因为 worker 重启后将无法用正确凭证恢复 provider job；应按现有 `provider_job_id` 持久化失败的 fail-fast 语义处理。

## 非功能需求

### 兼容性

- `credential_pool_enabled=false` 为默认值。
- 旧数据迁移后生成行为必须不变。
- 现有 active key UI 和调用路径在池化关闭时保持稳定。
- 旧任务没有 `credential_id` 绑定时，应走历史兼容路径。

### 部署与迁移

该需求涉及数据库模型变更，迁移必须保持默认关闭与旧行为兼容。

- 数据库迁移后，所有 provider 的 `credential_pool_enabled` 必须默认为 `false`。
- `provider_credential.is_enabled` 的迁移默认值需要在实现设计中明确二选一：全部默认为 `false`，或仅 `is_active=true` 的凭证默认为 `true`；无论选择哪种，由于 `credential_pool_enabled=false`，升级后生成行为不得改变。
- 新 lease/binding 表应允许空表启动，不要求回填历史任务。
- 旧 running/resume 任务没有 `credential_id` 时走兼容路径并记录 warning。
- 发布顺序建议：先发布数据库迁移和后端兼容字段，再发布前端池化开关。
- 回滚策略：关闭所有 provider 的 `credential_pool_enabled` 回到 active key 路径；已绑定任务仍需用绑定 credential 完成，不得因回滚改用 active 凭证。
- 需要新增 ADR：将 provider 凭证池化记录为 ADR 0016 的受开关保护扩展，明确“手动 active 凭证”仍是默认模式，池化是显式 opt-in 模式。

### 一致性

- 凭证租赁必须是原子操作。
- provider job 与 credential 绑定必须在 submit 成功后可靠持久化。
- 任务终态必须释放租赁。
- worker 崩溃后不得错误释放仍需 resume 的 provider job 绑定。

### 安全与密钥处理

池化层只处理凭证身份，不复制或暴露凭证明文。

- 租赁表、task payload、provider job binding 只保存 `credential_id`，不得复制 `api_key` / `access_key` / `secret_key`。
- 日志只允许输出 `provider_id`、`credential_id`、凭证名称、`provider_job_id` 等定位信息；不得输出明文 secret。
- `base_url` 可能含中转鉴权信息时，日志应避免完整输出或做必要脱敏。
- 前端池化 UI 只展示脱敏字段，与现有凭证列表一致。
- 连接测试失败返回的 provider 错误需要截断，并避免回显 secret。
- 删除、禁用、启用池化、切换参与池化状态等接口必须校验凭证属于指定 provider，不能跨 provider 操作 `credential_id`。

### 可观测性

### 性能与数据库访问

凭证池状态会进入 worker 高频 claim 路径，不能引入每任务/每 provider 的 N+1 查询。

- worker claim 前的凭证池状态应批量加载，或按 provider 缓存当前调度周期内的快照。
- active lease 查询不得扫描全表。
- 租赁表需要支持按 `provider_id`、`credential_id`、`status`、`media_type` 查询的索引。
- 租赁释放与恢复扫描需要 bounded batch，避免启动时或定时修复时持有长事务。
- SQLite 开发环境下需避免频繁 claim/requeue 导致写锁竞争；无空闲凭证时优先在 claim 前过滤。

日志应能说明：

- 任务租赁了哪个 provider 的哪条 credential id。
- 任务因无空闲凭证而等待。
- resume 使用了哪个 credential id。
- 删除凭证被拒绝的原因。
- 任务列表或任务详情应能暴露稳定等待原因 code，例如 `waiting_for_credential`，由前端 i18n 渲染。
- Provider 详情或诊断信息应能展示脱敏后的池化摘要：`credential_pool_enabled`、并发模式、参与池化凭证数量、active lease 数量。
- 单个任务详情应能展示绑定的凭证名称或脱敏标识，用于排查；不得展示明文 secret。
- 系统诊断日志下载应包含脱敏后的 credential lease 摘要，便于排查任务长期排队。

### 国际化

新增用户可见文案必须补齐 zh/en/vi：

- 启用凭证池化
- 参与池化
- 并发模式
- 图片/视频共享并发
- 图片/视频分别并发
- 等待可用凭证
- 凭证仍有关联运行中任务，无法删除

## 异常场景

1. 池化开启但没有任何 enabled 凭证：任务等待并提示 provider 无可用凭证。
2. 所有 enabled 凭证都被租赁：任务等待可用凭证。
3. 任务 submit 前租赁成功，但 submit 失败：任务按现有失败/重试语义处理，并在终态释放租赁。
4. submit 出现 ambiguous error：不得自动换 key 重新提交，避免重复扣费。
5. worker 崩溃后 lease 仍 active：系统恢复时应根据 task 状态决定保留、恢复或释放。
6. 用户禁用正在运行任务使用的凭证：任务继续使用该凭证直到终态。
7. 用户删除正在运行任务使用的凭证：拒绝删除。
8. 用户关闭 provider 池化时已有任务在运行：已绑定任务继续使用原凭证；新任务恢复 active 凭证路径。

## 测试矩阵

该需求的主要风险在并发租赁、恢复和 UI 状态切换。实现时至少覆盖以下自动化测试：

1. 池化关闭：仍只使用 active credential，其他 enabled/disabled 凭证不参与生成。
2. 池化开启 `shared`：同 provider 同时 4 个图片/视频混合任务时，4 条 key 各租一次，第 5 个保持 queued。
3. 池化开启 `separate`：同一 key 可同时租 1 个 image + 1 个 video，但不能租第 2 个 image。
4. claim 前过滤：所有 key 忙时任务不进入 running，不注册 `SlotTable` 占用。
5. claim 后竞态：租赁失败时任务 requeue，本轮 credential-full 黑名单生效。
6. submit 成功后绑定持久化失败：任务 fail-fast，不继续 poll。
7. resume：任务绑定 `credential_id` 后，重启恢复使用同一 credential。
8. 删除占用凭证：接口返回 409，前端展示可理解提示。
9. 禁用占用凭证：不影响运行中任务，新任务不再租它。
10. backend cache：不同 `credential_id` 不复用同一 backend。
11. UI：池化开关切换后凭证行从 active radio 切为“参与池化”开关；保存失败时回滚 UI 或刷新后端状态。

1. 默认升级后，所有 provider 仍只使用 active 凭证，现有生成流程不变。
2. Agnes 开启池化且有 4 条 enabled key，`shared` 模式下同时提交 4 个图片/视频混合任务时，4 个任务分别租赁 4 条不同凭证。
3. 第 5 个 Agnes 任务在 4 条 key 都忙时不 submit、不失败，显示或记录等待可用凭证。
4. 任一任务完成后释放凭证，第 5 个任务随后可租赁释放的凭证并开始执行。
5. 视频任务由 `key-1` submit 后，poll 与 resume 均使用 `key-1` 对应的 `credential_id`。
6. backend cache 不会让 `key-2` 任务复用 `key-1` 的 backend。
7. 池化关闭的 provider 不读取租赁池，仍走 active 凭证。
8. 开启池化的 provider 与未开启池化的 provider 可以同时正确生成任务，互不影响。
9. 禁用凭证后，该凭证不再接新任务，但已运行任务不被打断。
10. 删除仍有关联运行任务的凭证时，接口返回明确错误，前端展示提示。
11. `shared` 模式下，同一凭证不能同时跑图片和视频。
12. `separate` 模式下，同一凭证可以同时跑一个图片任务和一个视频任务。

## 待确认问题

1. 第一版是否只对预置供应商生效，自定义供应商暂不纳入？当前建议：是。
2. 新增凭证在池化开启时是否默认参与池化？当前建议：默认不参与，避免误用新 key。
3. “等待可用凭证”第一版是否复用 queued 状态并增加等待原因，而不是新增任务状态？当前建议：复用 queued 状态。
